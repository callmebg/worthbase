## Context

WorthBase 是一个基于 Expo SDK 57 的 React Native 个人财务 App（75 源文件，11,669 行代码），核心功能已稳定。项目使用 Claude AI 辅助开发（vibe coding），5 天内完成从原型到可用产品。已在小红书做过 3 篇推广尝试，有真实数据可用于复盘。

**双平台差异：**
- **CSDN**: 支持 `[TOC]` 自动生成、代码高亮、LaTeX 公式；百度搜索权重高，适合长尾 SEO
- **掘金**: 支持专栏系列、卡片链接；推荐算法 favor 高质量内容，适合社区口碑

**约束：**
- 代码片段必须从项目源码中真实提取，不能用伪代码
- 截图使用真实 App 截图（`docs/screenshots/` 已有 10 张）
- 双平台维护同一份源稿，发布时手动适配格式差异

## Goals / Non-Goals

**Goals:**
- **轨道 A（3 篇营销文章）**: 以 vibe coding 为主线，从技术/管理/推广三个角度触达不同读者群，带来 Star 和关注
- **轨道 B（8 篇技术文档）**: 深度展示核心算法和架构设计，作为技术参考，被第一篇营销文章链接引流
- 建立双平台格式适配流程
- 三篇营销文章之间建立自引链，互相导流

**Non-Goals:**
- 不做视频内容（只做图文）
- 不做英文内容（仅中文社区）
- 不做自动化发布（手动发布到两个平台）
- 不上架应用商店（仅 token 成本投入，开源分发）

## Decisions

### 1. 双轨制内容架构

**决策**: 3 篇营销文章 + 8 篇技术文档，两套内容独立成系列，通过交叉链接关联。

```
轨道 A — 营销文章（先写后连发）           轨道 B — 技术文档（持续发布）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━            ━━━━━━━━━━━━━━━━━━━━━━━━━━━━
篇1 技术角度 ←─── 导航链接 ────→         8 篇深度技术文章
篇2 项目管理角度                          (CSDN + 掘金独立系列)
篇3 推广复盘（仅掘金）
   ↓ 自引链 ↓
篇1→篇2→篇3 互相链接
```

**理由**: 营销文章追求传播力（短、有故事、有数字），技术文档追求深度（代码、算法、架构）。混在一起会两头不讨好。分开后各自聚焦，通过链接互补。

### 2. 文章源文件管理

**决策**: 在 `docs/articles/` 下按轨道分目录：

```
docs/articles/
├── marketing/          # 轨道 A: 3 篇营销文章
│   ├── 01-tech-angle.md
│   ├── 02-pm-angle.md
│   └── 03-growth-retro.md
└── technical/          # 轨道 B: 8 篇技术文档
    ├── 01-project-overview.md
    ├── 02-holding-cost.md
    ├── ...
    └── 08-design-system.md
```

**理由**: 物理隔离两套内容，避免混淆。截图和代码引用共享同一个项目资源。

### 3. 双平台格式适配

**决策**: 维护一份通用 Markdown 源文件，发布时手动调整：
- 目录：CSDN 加 `[TOC]`，掘金用编辑器自带
- 图片：统一 GitHub raw 链接
- 标签：CSDN 用 `#React Native #Expo #SQLite #AI编程`，掘金用话题标签
- 掘金发布后手动添加链接卡片

**理由**: 自动化工具维护成本高，手动调整每篇只需 5 分钟。

### 4. 轨道 A — 营销文章的叙事策略

**决策**: vibe coding 作为贯穿三篇的主线，但每篇侧重点不同：

| 篇 | vibe coding 角色 | 核心钩子 |
|---|---|---|
| 技术篇 | "怎么做的"方法论 | "600 元 + AI = 一个完整的 App" |
| 管理篇 | "5 天完成"的关键原因 | "周末 2 天原型 → 晚上 2 天修 bug → 1 天总结" |
| 推广篇 | "给独立开发者的建议" | "小红书 3231 曝光 vs 454 曝光，算法惩罚链" |

**代码展示策略（技术篇 1000 字 / 3 个功能）：**
- **持有成本**（~400 字）: 展示 `getStrategy()` 工厂 + 一个策略的 3 个方法，配四种方法对比表。这是教科书级 Strategy 模式应用。额外展示 `AmortizationRecommender` 的智能推荐作为产品思维亮点。
- **净资产趋势**（~400 字）: 展示 `downsamplePreservingExtrema()` 降采样（~20 行）+ `ProjectionCalculator` 线性回归（~15 行），配趋势图截图。视觉冲击力最强。
- **自动备份**（~200 字）: 展示 `AppState` 监听触发备份的 5 行代码。"本地优先的数据安全就是这么朴素"。

### 5. 轨道 B — 技术文档的叙事结构

**决策**: 按「全景 → 深入 → 实践」三段式组织：

| 阶段 | 文章 | 代码量 | 目的 |
|------|------|--------|------|
| 全景 | 项目总览 | 架构图 | 建立全局认知 |
| 深入 | 持有成本(841行)、本地优先(385行)、净资产可视化(2030行)、资产生命周期、账户余额 | 核心代码 | 展示技术深度 |
| 实践 | React Native DX、设计系统 | 可复用模式 | 给出可落地的经验 |

### 6. 发布节奏

**决策**:
- **轨道 A**: 3 篇全部写好后，Day 1/3/5 连发。掘金首发，CSDN 次日跟进
- **轨道 B**: 每周 2 篇（周二 + 周四），独立发布

**轨道 A 发布顺序与交叉链接：**

```
Day 1: 技术篇 ──→ "下一篇：5天项目管理复盘"
Day 3: 管理篇 ──→ "下一篇：小红书推广踩坑"  +  "回顾：技术实现"
Day 5: 推广篇 ──→ "回顾：技术篇 + 管理篇"（前情提要）
```

**理由**: 连发制造话题密度，自引链让每篇给其他篇导流。推广篇仅发掘金（独立开发者社区更契合复盘内容）。

### 7. 小红书复盘数据策略

**决策**: 使用完整漏斗数据，不修饰失败，展示算法惩罚链：

**完整数据（截至 7/19）：**

| 指标 | 篇1 (7/12) | 篇2 (7/14) | 篇3 (7/19) |
|------|-----------|-----------|-----------|
| 曝光 | 3,231 | 454 | 待观察 |
| 观看 | 559 | 65 | 39 |
| CTR | 16.0% | 13.2% | — |
| 互动总计 | 48 | 1 | 5 |

**5 个教训（每个有数据支撑）：**
1. "人"比"功能"有互动 — 篇1: 48互动 vs 篇2: 1互动
2. 具体数字 ≠ 共鸣 — 篇2: 13.2% CTR 但 0 评论（标题吸引人点进来，但内容没留住人）
3. 爆款标题吸引来的人 ≠ 目标用户 — 篇3 "大厂离职"吸引的是看八卦的人
4. **算法惩罚链** — 篇1→篇2 曝光骤降 86%（3231→454），篇2 的 0 互动导致篇3 几乎无曝光
5. 转化漏斗结构性断裂 — 小红书→GitHub 4 步断裂，559 观看仅 ~2-5 下载

**关键 insight**: CSDN/掘金文章的读者恰好是有能力完成 GitHub 下载路径的技术开发者。如果继续发小红书，应转为"真实使用记录"风格（净资产更新、资产卖出复盘、月度持有成本），不再做功能介绍。

**注意**: 篇3 发布不到 24h，数据待 48h 后更新再定稿。

## Risks / Trade-offs

- **[内容过时]** → 代码片段标注适用的 App 版本号，后续代码变更时同步检查
- **[写作耗时]** → 营销文章 ~2500 字/篇，技术文档 ~3000 字/篇，聚焦单主题
- **[图片托管]** → GitHub raw 链接有限流风险 → 备选：CDN 图床
- **[600 元追问]** → 准备成本分解表（仅 API token 费用），精确到元
- **[小红书数据不全]** → 篇3 刚发，数据待观察；文章中注明"截至发布时"的数据
- **[双轨维护成本]** → 轨道 B 的 8 篇文档周期较长（4-6 周），需持续投入；轨道 A 的 3 篇短期集中完成
