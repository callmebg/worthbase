# WorthBase（家底）计算引擎技术文档

> 本文档面向开发者，详细说明 WorthBase 计算引擎的架构、公式、边界情况及数据流。

---

## 一、架构总览

WorthBase 的计算引擎采用**策略模式（Strategy Pattern）** + **组合计算（Composable Calculators）** 的设计。每个计算器职责单一，通过组合实现复杂的财务计算。

### 核心模块

| 模块 | 职责 | 文件 |
|------|------|------|
| 分摊策略（4种） | 计算购入价的月度分摊 | `strategies/` |
| 持有成本计算器 | 汇总分摊 + 经常性支出 + 维护 | `HoldingCostCalculator.ts` |
| 净资产计算器 | 账户余额 + 资产估值 | `NetWorthCalculator.ts` |
| 卖出结算计算器 | 卖出时的财务结算 | `SettlementCalculator.ts` |
| 经常性支出计算器 | 按月生效的周期性支出 | `RecurringExpenseCalculator.ts` |
| 维护费计算器 | 一次性维护支出的分摊 | `MaintenanceCalculator.ts` |
| 净资产预测计算器 | 线性回归预测目标达成日期 | `ProjectionCalculator.ts` |
| 分摊推荐引擎 | 根据资产类别推荐分摊方式 | `AmortizationRecommender.ts` |

## 三、分摊策略（Amortization Strategies）

分摊策略决定了资产购入价如何分配到每个月。系统提供 4 种策略，通过**策略工厂**（`getStrategy(asset)`）根据资产的 `amortizationType` 字段自动选择。

### 策略接口

每个策略实现 3 个方法：

```typescript
interface AmortizationStrategy {
  calculateMonthlyCost(asset, currentDate): number;   // 当月分摊金额
  calculateAccumulated(asset, currentDate): number;   // 累计已分摊金额
  calculateRemaining(asset, currentDate): number;     // 剩余未分摊金额
}
```

### 公共工具函数

#### `monthsBetween(startDate, endDate)`

计算两个日期之间的月数（含起始月），最小返回 1。

**规则：**
- 如果结束日的「日」≥ 起始日的「日」，则 +1（满整月）
- 否则 +0（不满整月，但起始月仍计为 1）
- 最终结果 `Math.max(1, months)`

**示例：**
```
2025-01-15 → 2025-07-15：6个月（15 ≥ 15，+1，结果 6+1=7？不对）
实际计算：yearDiff=0, monthDiff=6, date 15 >= 15 → months = 6+1 = 7
但代码中：months = 0*12 + 6 = 6, 然后 15 >= 15 → months = 6+1 = 7

2025-01-15 → 2025-01-16：monthDiff=0, date 16 >= 15 → months = 0+1 = 1

2025-01-15 → 2025-01-14：monthDiff=0, date 14 < 15 → months = 0+0 = 0 → Math.max(1,0) = 1
```

#### `daysBetween(startDate, endDate)`

计算两个日期之间的天数，最小返回 1。

```
days = Math.max(1, Math.ceil((endDate - startDate) / (1000 × 60 × 60 × 24)))
```

---

### 策略 1：简单线性分摊（Simple Linear）

**适用场景：** 电子产品、数码产品等快速贬值资产

**核心思想：** 购入价均摊到已持有的所有月份。持有越久，月均成本越低。

**公式：**

```
月分摊成本 = 购入价 ÷ 已持有月数

累计已分摊 = 购入价（始终全额分摊）

剩余未分摊 = 0
```

**示例：** 一台 iPhone 购入价 ¥8,000

| 持有时间 | 月分摊成本 | 累计已分摊 | 剩余未分摊 |
|---------|-----------|-----------|-----------|
| 1 个月  | 8000 ÷ 1 = ¥8,000 | ¥8,000 | ¥0 |
| 6 个月  | 8000 ÷ 6 = ¥1,333 | ¥8,000 | ¥0 |
| 12 个月 | 8000 ÷ 12 = ¥667 | ¥8,000 | ¥0 |
| 24 个月 | 8000 ÷ 24 = ¥333 | ¥8,000 | ¥0 |

**边界情况：**
- `purchasePrice ≤ 0` → 月分摊 = 0
- 已持有月数始终 ≥ 1，不会出现除以零

---

### 策略 2：预期寿命分摊（Expected Lifespan）

**适用场景：** 房产、家居用品等有明确使用寿命的资产

**核心思想：** 购入价均摊到预期寿命的每个月。月分摊金额固定不变。

**公式：**

```
月分摊成本 = 购入价 ÷ 预期寿命月数（固定值）

累计已分摊 = 月分摊成本 × min(已持有月数, 预期寿命月数)
           = min(累计值, 购入价)

剩余未分摊 = max(0, 购入价 - 累计已分摊)
```

**示例：** 一套沙发购入价 ¥12,000，预期寿命 108 个月（9 年）

| 持有时间 | 月分摊成本 | 累计已分摊 | 剩余未分摊 |
|---------|-----------|-----------|-----------|
| 1 个月  | 12000 ÷ 108 = ¥111.11 | ¥111.11 | ¥11,888.89 |
| 12 个月 | ¥111.11 | ¥1,333.33 | ¥10,666.67 |
| 108 个月| ¥111.11 | ¥12,000 | ¥0 |
| 120 个月| ¥111.11 | ¥12,000（封顶）| ¥0 |

**边界情况：**
- `expectedLifespanMonths` 为 null 或 ≤ 0 → 月分摊 = 0，剩余 = 购入价
- 持有超过预期寿命后，累计封顶为购入价，剩余归零

---

### 策略 3：残值分摊（Residual Value）

**适用场景：** 车辆等在使用寿命结束后仍有残值的资产

**核心思想：** 只分摊「会贬值的部分」，即购入价减去预估残值。

**公式：**

```
可分摊金额 = 购入价 - 预估残值

月分摊成本 = 可分摊金额 ÷ 预期寿命月数（固定值）

累计已分摊 = 月分摊成本 × min(已持有月数, 预期寿命月数)
           = min(累计值, 可分摊金额)

剩余未分摊 = max(0, 购入价 - 累计已分摊)
```

> **注意：** 剩余未分摊使用 `购入价 - 累计已分摊`，而不是 `可分摊金额 - 累计已分摊`。
> 这意味着当分摊完成后，剩余 = 残值金额，而非零。

**示例：** 一辆车购入价 ¥200,000，预估残值 ¥50,000，预期寿命 60 个月（5 年）

```
可分摊金额 = 200,000 - 50,000 = ¥150,000
月分摊成本 = 150,000 ÷ 60 = ¥2,500
```

| 持有时间 | 月分摊成本 | 累计已分摊 | 剩余未分摊 |
|---------|-----------|-----------|-----------|
| 1 个月  | ¥2,500 | ¥2,500 | ¥197,500 |
| 12 个月 | ¥2,500 | ¥30,000 | ¥170,000 |
| 60 个月 | ¥2,500 | ¥150,000 | ¥50,000 |
| 72 个月 | ¥2,500 | ¥150,000（封顶）| ¥50,000 |

**边界情况：**
- `residualValue` 为 null 时按 0 处理（退化为预期寿命分摊）
- `可分摊金额 ≤ 0`（残值 ≥ 购入价）→ 月分摊 = 0
- `expectedLifespanMonths` 为 null 或 ≤ 0 → 月分摊 = 0，剩余 = 购入价

---

### 策略 4：不分摊（No Amortization）

**适用场景：** 奢侈品、贵金属等可能增值的资产

**核心思想：** 购入价不计入月度成本。持有成本仅由经常性支出和维护费构成。

**公式：**

```
月分摊成本 = 0

累计已分摊 = 0

剩余未分摊 = 购入价（永远不减少）
```

**示例：** 一块金条购入价 ¥50,000

| 持有时间 | 月分摊成本 | 累计已分摊 | 剩余未分摊 |
|---------|-----------|-----------|-----------|
| 任意时间 | ¥0 | ¥0 | ¥50,000 |

---

`AmortizationRecommender` 根据资产类别自动推荐分摊方式：

| 资产类别 | 推荐策略 | 默认寿命 | 推荐理由 |
|---------|---------|---------|---------|
| 电子产品 | 简单线性 | — | 贬值快，按已持有时间递减 |

| 其他 | 简单线性 | — | 默认简单线性 |

## 四、持有成本计算器（HoldingCostCalculator）

### 概述

持有成本是 WorthBase 的核心指标，回答一个关键问题：**拥有一件资产，每个月实际要花多少钱？**

### 公式

```
月持有成本 = 月分摊成本 + 月经常性支出 + 月维护分摊

日均持有成本 = 月持有成本 ÷ 30

累计持有成本 = 累计分摊 + 累计经常性支出 + 累计维护分摊（含非分摊部分）
```

### 各分项详解

#### 1. 月分摊成本（monthlyAmortization）

由所选的分摊策略计算，详见第三节。

#### 2. 月经常性支出（monthlyRecurring）

当月生效的所有经常性支出之和。

```
月经常性支出 = Σ(当月生效的经常性支出项.amount)
```

**示例：** 一辆车的经常性支出：
- 车险：¥500/月（2025-01 起生效）
- 保养套餐：¥200/月（2025-03 起生效，2025-12 结束）
- 停车费：¥300/月（2025-01 起生效）

2025-04 的月经常性支出 = 500 + 200 + 300 = ¥1,000
2026-01 的月经常性支出 = 500 + 300 = ¥800（保养套餐已结束）

#### 3. 月维护分摊（monthlyMaintenance）

一次性维护费用可选择「纳入分摊」或「不纳入」。纳入分摊的维护费按剩余持有月数均摊。

**分摊逻辑：**

```
对于每条纳入分摊的维护记录：

  如果资产有预期寿命：
    已持有至维护时的月数 = monthsBetween(购入日, 维护日)
    剩余月数 = max(1, 预期寿命月数 - 已持有至维护时的月数)

  如果资产无预期寿命：
    剩余月数 = max(1, monthsBetween(维护日, 当前日))

  月分摊 = 维护金额 ÷ 剩余月数

月维护分摊 = Σ(各分摊维护记录的月分摊)
```

**示例：** 一辆车预期寿命 60 个月，在第 12 个月做了一次 ¥6,000 的维修（纳入分摊）

```
剩余月数 = max(1, 60 - 12) = 48
月分摊 = 6000 ÷ 48 = ¥125/月
```

如果这辆车没有预期寿命，且维修发生在 6 个月前：

```
剩余月数 = max(1, 6) = 6
月分摊 = 6000 ÷ 6 = ¥1,000/月
```

### 完整示例

一台 MacBook Pro，购入价 ¥15,000，简单线性分摊，已持有 10 个月：

| 分项 | 计算 | 金额 |
|------|------|------|
| 月分摊成本 | 15000 ÷ 10 | ¥1,500 |
| 月经常性支出 | AppleCare ¥99 + iCloud ¥21 | ¥120 |
| 月维护分摊 | ¥0（无维护记录） | ¥0 |
| **月持有成本** | | **¥1,620** |
| **日均持有成本** | 1620 ÷ 30 | **¥54** |

### 边界情况

- **非活跃资产（retired / sold）：** 所有持有成本归零，返回全零结果
- **批量计算：** `calculateAll()` 遍历所有资产逐个计算，返回 `Map<assetId, HoldingCostResult>`
- **总计：** `getTotalMonthly()` 只统计 `status === ACTIVE` 的资产

## 五、经常性支出计算器（RecurringExpenseCalculator）

### 概述

经常性支出是附着在资产上的周期性费用（如保险、订阅、话费等），每条记录有生效区间 `[effectiveFrom, effectiveTo]`。

### 月度汇总

```
getMonthlyTotal(month, assetId?) = Σ(当月生效的所有支出项.amount)
```

### 累计经常性支出

```
getAccumulatedTotal(assetId, purchaseDate, currentMonth):

  对每条支出项：
    起始月 = max(expense.effectiveFrom, purchaseMonth)
    结束月 = min(expense.effectiveTo ?? currentMonth, currentMonth)

    如果 起始月 ≤ 结束月：
      生效月数 = monthDiff(起始月, 结束月) + 1
      累计 += amount × 生效月数

  返回累计值
```

**`monthDiff` 函数：**

```
monthDiff(from, to) = (toYear - fromYear) × 12 + (toMonth - fromMonth)
```

### 示例

一辆车购入于 2025-01，当前为 2025-06：

| 支出项 | 金额 | 生效起 | 生效止 | 生效月数 | 累计 |
|--------|------|--------|--------|---------|------|
| 车险 | ¥500 | 2025-01 | null | 6 | ¥3,000 |
| 保养 | ¥200 | 2025-03 | 2025-05 | 3 | ¥600 |
| 停车 | ¥300 | 2025-02 | null | 5 | ¥1,500 |
| **合计** | | | | | **¥5,100** |

## 六、维护费计算器（MaintenanceCalculator）

### 概述

一次性维护费用（如换电池、维修、翻新）分为两类：
- **纳入分摊（amortize = true）：** 按剩余月数均摊到每月持有成本中
- **不纳入分摊（amortize = false）：** 仅记录，不影响月持有成本，但计入累计总成本

### 核心方法

#### `getAmortizedMonthly(asset, currentDate)`

计算当月应分摊的维护费总额（仅含 `amortize = true` 的记录）。

公式详见第四节「月维护分摊」部分。

#### `getNonAmortizedTotal(assetId)`

```
= Σ(所有 amortize = false 的维护记录.amount)
```

#### `getAccumulatedAmortized(asset, currentDate)`

计算分摊类维护费的累计已分摊金额：

```
对每条分摊记录：
  月分摊 = amount ÷ 剩余月数（计算方式同月维护分摊）
  已过月数 = monthsBetween(维护日, 当前日)
  累计 += 月分摊 × 已过月数
```

**示例：** 维护费 ¥6,000，剩余 48 个月，已过 6 个月：

```
月分摊 = 6000 ÷ 48 = ¥125
累计 = 125 × 6 = ¥750
```

#### `getAccumulatedTotal(assetId)`

```
= Σ(所有维护记录的 amount)（含分摊和非分摊）
```

## 七、净资产计算器（NetWorthCalculator）

### 概述

净资产回答「我现在值多少钱」这个核心问题。

### 公式

```
流动资产 = Σ(所有账户的最新余额)

资产估值 = Σ(开启估值追踪的活跃资产的最新估值)

净资产 = 流动资产 + 资产估值

未分摊成本 = Σ(活跃资产的剩余未分摊购入价)  // 内部参考，不展示给用户
```

### 示例

| 账户 | 余额 |
|------|------|
| 微信 | ¥5,000 |
| 支付宝 | ¥12,000 |
| 银行卡 | ¥80,000 |
| 信用卡 | -¥3,000 |
| **流动资产** | **¥94,000** |

| 资产 | 估值 | 是否追踪 |
|------|------|---------|
| 车辆 | ¥120,000 | ✅ |
| MacBook | ¥8,000 | ✅ |
| 金条 | ¥52,000 | ❌ |
| **资产估值** | **¥128,000** | |

```
净资产 = 94,000 + 128,000 = ¥222,000
```

### 目标进度

```
进度百分比 = (当前净资产 ÷ 目标金额) × 100%

是否达标 = 进度百分比 ≥ 100%
```

**边界情况：**
- 目标为 null 或 ≤ 0 → 进度 = 0%，未达标
- 信用卡和贷款账户的余额为负数（负债），自动扣减流动资产

## 八、卖出结算计算器（SettlementCalculator）

### 概述

当资产被卖出时，结算计算器提供一份完整的「财务账单」，告诉用户这件资产从买到卖到底花了多少钱。

### 公式

```
贬值 = 购入价 - 卖出价
  （如果资产增值，贬值为负数）

累计持有成本 = 累计分摊 + 累计经常性支出 + 累计维护费（含分摊与非分摊）

净支出 = 购入价 + 累计持有成本 - 卖出价

持有天数 = daysBetween(购入日, 卖出日)

日均成本 = 净支出 ÷ 持有天数
```

### 完整示例

一辆车的完整生命周期：

| 项目 | 值 |
|------|-----|
| 购入价 | ¥200,000 |
| 卖出价 | ¥120,000 |
| 持有时间 | 2024-01-15 → 2026-07-10（877天） |
| 分摊方式 | 残值分摊（残值 ¥50,000，寿命 60月） |
| 累计分摊 | ¥43,750（月分摊 ¥2,500 × 17.5月≈18月） |
| 累计经常性支出 | 车险 ¥500×18 + 停车 ¥300×18 = ¥14,400 |
| 累计维护费 | 保养 ¥2,000（非分摊） + 维修 ¥6,000（分摊累计 ¥1,500） = ¥8,000 |

计算结果：

```
贬值 = 200,000 - 120,000 = ¥80,000

累计持有成本 = 43,750 + 14,400 + 8,000 = ¥66,150

净支出 = 200,000 + 66,150 - 120,000 = ¥146,150

日均成本 = 146,150 ÷ 877 = ¥166.65
```

> 解读：这辆车开了两年半，总共花了 ¥146,150，平均每天花 ¥166.65。

### 预览模式（Preview）

`preview(asset, hypotheticalSellPrice)` 可以在不实际卖出资产的情况下，模拟卖出结算。内部构造一个临时资产对象（状态设为 sold，卖出日期为当前），然后调用 `calculate()`。

### 边界情况

- `sellDate` 为 null → 默认使用当前时间
- `sellPrice` 为 null → 默认 0（白送或丢弃）
- 卖价高于购入价时，贬值为负数（资产增值），但净支出仍可能为正（持有成本的影响）

## 九、净资产预测计算器（ProjectionCalculator）

### 概述

基于历史净资产数据点，使用**简单线性回归**预测何时能达成净资产目标。

### 算法

#### 输入

- `historicalPoints`：历史数据点数组 `[{date: 'YYYY-MM-DD', value: number}]`
- `goal`：目标净资产值

#### 步骤

**1. 取最近 6 个数据点**

```
points = historicalPoints.slice(-6)
```

**2. 简单线性回归计算月均增长率**

将日期转换为「距第一个数据点的月数」作为 x 轴，净资产值作为 y 轴：

```
x_i = (date_i - date_0) / (30.44 × 24 × 60 × 60 × 1000)  // 月数（30.44天/月）
y_i = value_i
```

使用最小二乘法求斜率：

```
slope = (n × Σ(x_i × y_i) - Σx_i × Σy_i) / (n × Σ(x_i²) - (Σx_i)²)
```

**3. 预测达成日期**

```
gap = goal - currentValue
monthsNeeded = gap / slope
achievementDate = lastDate + monthsNeeded × 30.44 天
```

### 示例

假设最近 6 个月的净资产数据：

| 月份 | 净资产 | x（月数） |
|------|--------|----------|
| 2025-01 | ¥180,000 | 0 |
| 2025-02 | ¥185,000 | 1 |
| 2025-03 | ¥189,000 | 2 |
| 2025-04 | ¥195,000 | 3 |
| 2025-05 | ¥200,000 | 4 |
| 2025-06 | ¥206,000 | 5 |

目标：¥300,000

```
n = 6
Σx = 0+1+2+3+4+5 = 15
Σy = 180000+185000+189000+195000+200000+206000 = 1,155,000
Σxy = 0×180000 + 1×185000 + 2×189000 + 3×195000 + 4×200000 + 5×206000
    = 0 + 185000 + 378000 + 585000 + 800000 + 1030000 = 2,978,000
Σx² = 0+1+4+9+16+25 = 55

slope = (6 × 2978000 - 15 × 1155000) / (6 × 55 - 15²)
      = (17868000 - 17325000) / (330 - 225)
      = 543000 / 105
      ≈ ¥5,171/月

gap = 300000 - 206000 = ¥94,000
monthsNeeded = 94000 / 5171 ≈ 18.2 个月

预计达成日期 ≈ 2025-06 + 18.2月 ≈ 2026-12
```

### 返回 `null` 的情况

- 数据点少于 2 个
- 所有数据点 x 值相同（无法做回归）
- 斜率 ≤ 0（净资产没有增长或在下降）
- 当前净资产已经 ≥ 目标

## 十、关键设计决策

### 1. 日均按 30 天计算

月持有成本转换为日均时统一除以 30，不考虑大小月差异。这保证了月度指标和日均指标之间的一致性。

### 2. 简单线性策略的「全额分摊」特性

简单线性策略下，`accumulated = purchasePrice`，`remaining = 0`。这意味着购入价被视为「已经全部花出去了」，只是随着时间推移月均成本在降低。这与预期寿命策略不同——后者有明确的「分摊完毕」时间点。

### 3. 非分摊维护费的双重角色

- **不影响月持有成本**：不纳入分摊的维护费不会增加月持有成本
- **影响累计总成本**：在 `accumulatedTotal` 中，非分摊维护费被直接加入
- **影响卖出结算**：在 `SettlementCalculator` 中，所有维护费（分摊与非分摊）都计入持有成本

### 4. 策略工厂的回退机制

如果资产的 `amortizationType` 字段值无法识别（理论上不会发生，但做了防御），自动回退到 `NoAmortizationStrategy`，保证不会崩溃。

### 5. 日期处理

- 所有日期存储为 ISO 8601 格式
- 月份比较使用 `YYYY-MM` 字符串直接比较（字典序 = 时间序）
- 月数计算使用 `30.44 天/月` 的平均值（年 365.25 天 ÷ 12）
