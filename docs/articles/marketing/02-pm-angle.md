# 一个人从 0 到 1 做一个 App，我是怎么用 vibe coding 在 5 天内管理这个项目的

一个人做的。5 天。

## 需求管理：只做自己需要的功能

独立开发最大的陷阱是**功能蔓延**。你不是在做产品，你是在满足自己的强迫症。

我做需求管理的原则很简单：**这个功能是我自己日常会用的吗？** 如果不是，砍掉。

### 砍掉的三个功能

**1. 自动记账**

几乎所有记账 App 都有这个功能——连接银行卡、自动识别消费记录。技术上需要对接银行 API 或者短信解析，复杂度高，而且和"隐私优先"的定位直接矛盾。你要自动记账，就得读取用户的短信或者授权银行接口，那还谈什么"不联网不上传"？

砍掉。手动更新余额就够了。我要的是"财务状态快照"，不是"每笔流水记录"。

**2. 预算功能**

"这个月餐饮预算 2000，已经花了 1500"——这种功能在专门的记账 App 里很好，但不是我要解决的问题。我关心的是"我的净资产在涨还是在跌"，不是"这个月外卖花了多少"。

砍掉。不做流水账，只做状态管理。

**3. 网络同步**

多设备同步是很多用户提的需求。但一旦做了云同步，就要做用户系统、数据加密、冲突解决、服务端运维……整个项目的复杂度会翻三倍。

砍掉。本地优先就是本地优先。需要换设备？导出 JSON 文件，新设备导入，完事。

### 砍掉之后的效果

砍掉这三个功能，项目范围缩小了大约 40%。更重要的是，**产品定位变得极其清晰**：不做记账 App，做财务状态管理 App。不记流水，只看快照。

小而美，不贪大求全。

不是先做完再推广，而是边做边推。小红书发出去的那一刻，你就有了真实用户反馈。

## Vibe Coding 改变了单人开发效率

传统流程：想需求 → 写代码 → 调试 → 测试 → 修 Bug

Vibe Coding 流程：**描述需求 → AI 生成代码 → 跑起来 → 描述 Bug 让 AI 改**

举个例子。我要做"净资产趋势的降采样算法"，传统做法是：
1. 研究降采样算法（LTTB、Min-Max 等）
2. 选一种实现
3. 写代码、调试、处理边界情况
4. 可能花半天到一天

Vibe Coding 的做法：
1. 告诉 Claude："数据点超过 24 个时需要降采样，但要保留峰值和谷值，不能被等距取样抹平"
2. Claude 产出 `downsamplePreservingExtrema()` 函数，20 行
3. 跑起来，效果满意
4. 总共花 10 分钟

**效率差异的核心不在于 AI 写代码更快，而在于 AI 帮你跳过了"从 0 到 0.8"的重复劳动。** 你只需要做最后 20% 的审核和调优——而那 20% 恰恰是最需要人类判断的部分。

## 一个人怎么做测试

没有 QA 团队，没有测试环境，独立开发者的测试策略必须务实：

**第一层：Jest 单元测试（2,800+ 行，8 个测试文件）**

只测核心引擎——4 种分摊策略的数学计算、卖出结算公式、投影计算。这些是纯函数，没有 UI 依赖，最适合单元测试。

```typescript
// 测试 SimpleLinearStrategy：月均成本应该随持有时间递减
test('monthly cost decreases over time', () => {
  const cost6m = strategy.calculateMonthlyCost(asset, new Date('2025-07-15'));
  const cost12m = strategy.calculateMonthlyCost(asset, new Date('2026-01-15'));
  expect(cost6m).toBeCloseTo(1714.29, 1); // 12000 / 7
  expect(cost12m).toBeCloseTo(923.08, 1);  // 12000 / 13
  expect(cost6m).toBeGreaterThan(cost12m);
});
```

**第二层：设备手动测试**

Expo Go 扫码就能在真机上跑。热更新让"改代码→看效果"的循环缩短到秒级。关键路径手动走一遍：添加账户 → 更新余额 → 添加资产 → 查看趋势图 → 导出数据。

**第三层：小红书用户反馈（最意想不到的测试层）**

第一篇小红书帖子有 18 条评论。其中 2 条帮我发现了真实的 Bug——一个关于负余额显示的 UI 问题，一个关于日期格式的问题。这些是我自己测试时没覆盖到的场景，因为我太熟悉自己的 App 了，不会"正常使用"它。

**结论：对独立开发者来说，真实用户是最好的 QA。**

## 总结：独立开发者项目管理的 3 个关键点

**1. 用"我自己会不会用"作为需求过滤器**

砍掉自动记账、预算功能、网络同步，不是因为做不了，而是因为它们不属于"我需要的 App"。每多做一个人需求，就多一份维护成本。

**2. 先出原型再做推广，用真实反馈驱动迭代**

不要等"做完了"再推广。Day 2 就发小红书，Day 3 就有真实用户反馈。你的用户就是你的免费 QA。

**3. 让 AI 做 80% 的重复劳动，你做 20% 的产品决策**

Vibe Coding 不是"让 AI 替你思考"，而是"让 AI 替你搬砖，你负责指挥方向"。AI 不知道"持有成本怎么算才符合直觉"，但你知道。这才是独立开发者的核心价值。
